# kiro.rs 并发调度中心实施计划

> 本文是 [并发调度中心迁移方案.md](./并发调度中心迁移方案.md) 的落地补充。
> 方案文档讲“为什么这么改”，本文讲“具体改哪些文件、按什么顺序、怎么验”。

## 定位说明

kiro.rs 是 junnl 的升级版，junnl 的并发调度中心是已在生产用过一段时间、验证过的
自有二开资产。因此本文所说的“迁移”不是两个并列项目之间的移植，而是把 junnl 上
**已验证好的调度内核原样上拉进它的升级版 kiro.rs**。

据此明确两条原则：

- **junnl 调度中心只读、不改**：仅作为算法来源参考，所有改动都落在 kiro.rs。
- **算法照搬、过滤叠加**：junnl 的选号算法（active 计数、RAII guard、least-active、
  会话粘性、满载短等）一行不改地拉过来；kiro.rs 升级版已有的 `throttled_until`
  风控冷却与 `groups` 分组隔离，作为**额外候选过滤条件叠加进选号**，不动 junnl
  内核逻辑。

## 范围确认

一次性完成四个阶段：核心调度 → 并发硬上限 → 会话粘性 → 管理面板。
收尾必须 `cargo test` 与 `npm run build` 双绿。

迁移边界（与方案一致，不重复展开）：

- 只改账号调度、并发槽位、会话粘性、管理面板展示与设置。
- 不碰 provider 请求转换、模拟缓存、usage 改写、请求日志真实字段、429/402/401 判断语义。
- 并发计数只做单实例内存版，不上 Redis。

## 关键设计决策

### 1. 会话粘性采用 junnl 单层版，不移植 sub2 多层

调研结论（核实 sub2api 原版后确认）：

- **sub2 原版**是与窗口费用 / RPM 配额深度绑定的多层调度：三区模型
  （`Schedulable` / `StickyOnly` / `NotSchedulable`）、RPM 粘性缓冲
  （`GetRPMStickyBuffer`）、粘性预留额度（`GetWindowCostStickyReserve`）、
  `sticky_exempt` 豁免策略等多层。
- **junnl 已砍到单层**：只保留“会话绑账号 → 绑定号可用就复用 → 满载短等 2s →
  等待者 ≥2 放弃亲和换号”。junnl 注释里写明 `STICKY_MAX_WAITING=2` 是
  “远小于 sub2 的 3”。
- **kiro.rs 决定照搬 junnl 单层版**。原因：kiro.rs 没有 sub2 的费用 / RPM 配额
  基础设施（无 window cost、无 RPM 三区），硬移植 sub2 多层粘性等于要先迁一整套
  配额体系，远超方案边界；方案文档也明确推荐“轻量内存版而非复杂队列”。

### 2. 保留 kiro.rs 全部独有逻辑（方案风险点 6）

junnl 原版没有、但 kiro.rs 现有、迁移时绝不能漏的字段：

- `throttled_until`：账号级 429 风控临时冷却。
- `groups` + `group_matches`：账号分组隔离。
- `total_failure_count`：累计失败统计。

**选号过滤必须同时排除**：`disabled` + 风控冷却中 + opus 不支持 + 分组不匹配。
这是 kiro.rs 版 `reserve_best` 与 junnl 版的核心差异点。

### 3. RAII guard 释放，禁止手动维护 active

`active` 只在选中时同锁 `+1`，释放只能走 `ConcurrencyGuard::drop`。
覆盖全部分支：成功 / 429 退避 / 402 切号 / 网络错误 / 客户端断开 / 流结束 / panic。

### 4. 性能铁律：绝不拖慢「上游返回 → 用户」的响应速度

并发调度只准在两个点干活，且这两个点都**不在**「上游开始返回 → 用户收完」这段
热路径里：

```
[请求进来]
  → acquire_context   ← 调度干活点①：选号 + active+=1（在上游请求之前）
  → 发上游请求
  → 上游开始返回 ──────────────┐
  → 流式逐字吐给用户            │  ← 这段管道：调度零参与、零开销
  → 流读完 ─────────────────────┘
  → guard Drop        ← 调度干活点②：锁 + active-=1（微秒级，不在传输路径上）
```

硬约束（写代码时不可违反）：

1. **guard 持有期零开销**：slot_guard 装进 `StreamContext` / `_slot_guard` 后，在
   响应流转过程中**只是被 move 传递，不得有任何锁、await、per-chunk 计算**。它只在
   `Drop` 那一刻做一次 `saturating_sub(1)`。首字延迟、吐字速度完全不受影响。
2. **选号锁内禁 await / 禁 IO / 禁重计算**：`entries` 锁内只做「遍历候选 + 算术 +
   active+=1」，微秒级。任何 sleep / 网络 / token 刷新都必须在释放锁之后做
   （与风险点 3「锁顺序死锁」同源约束）。
3. **默认配置零等待**：满载短等（2s）与粘性等待只在 `maxConcurrency>0 且真满载` 时
   触发。默认 `maxConcurrency=0`（不限制）下 `reserve_best` 永不判满载，第一次即
   选中返回，**一次 sleep 都不会发生**。即“不主动配并发上限的用户，全程无新增等待”。
4. **release 时机不影响感知速度**：guard Drop 是纯后台动作（一次锁 + 算术）。
   - 流式：用户收完 / 连接结束后释放。
   - 非流式：上游 body 读完并完成响应构造后、handler 返回前释放（此时客户端未必已
     收到，但 drop 本身只是微秒级算术，不在响应传输路径上、不增加传输延迟）。

唯一需盯防的是 `entries` 锁竞争（selection / drop / report_* / snapshot 共用一把
parking_lot Mutex）。但临界区都是无 IO 无 await 的极短操作，且这把锁现在就存在，
不是新增瓶颈——只要守住上面 1、2 两条铁律即可。

**验收**：并发压测对比「开调度 vs 关调度」的首字延迟 P50/P99 与整体吞吐，
确认无可见回归（详见测试清单）。

## 阶段一：核心调度

文件：`src/kiro/token_manager.rs`、`src/main.rs`

### token_manager.rs

1. `CredentialEntry` 新增运行时字段（不持久化）：
   - `active: u32`：当前在途请求数，选中即 `+1`，guard drop 时 `-1`。
   - `waiting: u32`：等待该号释放的请求数，仅前端展示与粘性过载判断。
   - `new()` / `add_credential()` 两处初始化都要补 `active: 0, waiting: 0`。

2. 新增 `ConcurrencyGuard`：
   ```rust
   pub struct ConcurrencyGuard {
       manager: Weak<MultiTokenManager>,
       credential_id: u64,
       released: bool,
   }
   ```
   - `new` / `noop` 构造，`Drop` 调 `manager.upgrade()?.release_active_slot(id)`。

3. `MultiTokenManager` 新增字段：
   - `self_weak: Mutex<Weak<MultiTokenManager>>`
   - `session_affinity: Mutex<HashMap<String, SessionAffinityEntry>>`
   - `SessionAffinityEntry { credential_id: u64, last_used_at: Instant }`

4. 新增方法：
   - `init_weak_self(self: &Arc<Self>)`：填充 self_weak。
   - `make_guard(id) -> ConcurrencyGuard`：self_weak 未初始化时返回 noop。
   - `release_active_slot(id)`：持 entries 锁 `saturating_sub(1)`。
   - `reserve_best(model, group) -> ReserveOutcome`：least-active 选号，**过滤保留
     throttled/group/opus**，同锁内 `active += 1`。
   - `reserve_one(id, model, group) -> ReserveOne`：占用指定号（粘性命中用）。
   - `reserve_best_with_wait(model, group)`：满载短等 + 自愈。
   - `wait_for_sticky(bound_id, model, group)`：粘性原号满载短等。
   - `reserve_context(model, group, session_key)`：选号总入口。
   - `adjust_waiting(id, inc)` / `waiting_count(id)`。
   - `credential_for_session` / `remember_session_affinity` / `clear_session_affinity`
     / `clear_session_affinity_for_credential` / `clear_all_session_affinity`。
     - `clear_session_affinity_for_credential(id)` 调用时机（与 junnl 成熟版一致）：
       **凡是“账号变不可用”或“groups 改变”的方法都要主动清绑**，避免会话粘到已
       失效的号。当前仓库需覆盖的全部触发点：
       - 变不可用：`report_failure`（自动禁用）、`report_quota_exhausted`、
         `report_refresh_failure`（自动禁用）、`report_refresh_token_invalid`、
         `disable_quota_exceeded`、手动禁用、删除凭据、账号级风控冷却
         （`report_account_throttled`）。
       - 恢复/变更：`reset_and_enable`、`update_refresh_token`（重新登录换号）。
       - groups 改变：`rename_credential_group`、`remove_credential_group`、
         单账号改组。组改了用 `clear_all_session_affinity` 或按受影响号清。
       注：`reserve_one` 命中不可用号时也会自愈清绑，故漏一两处不致错，但 junnl 是
       主动清，**强烈建议逐一对齐**，否则粘性会短暂指向坏号、增加一次无效 reserve。

5. 选号策略：
   - **balanced**：主键 `active` 最小 → 平手 `success_count` 最小 → 再平手组内随机。
   - **priority**：先锁定最高优先级（priority 最小）**可用**档 → 档内 `active` 最小
     → 平手随机。**不可写成 `(active, priority)`**，否则低优先级空闲号会抢高优先级。

6. `acquire_context` 改为“一薄封装 + 一新方法”，参数不变，返回值带上 guard：
   - `acquire_context(model, group)`：**参数保持不变**，但返回值由 `CallContext`
     改为 `(CallContext, ConcurrencyGuard)`。内部实现为
     `acquire_context_for_session(model, group, None)` 的薄封装。
   - **新增** `acquire_context_for_session(model, group, session_key)
     -> (CallContext, ConcurrencyGuard)`：带会话粘性的真正实现。
   - ⚠️ **acquire 内部刷新失败必须 drop guard（最关键正确性点）**：当前
     [token_manager.rs:1273-1294](../src/kiro/token_manager.rs#L1273) 在选号后还会
     `try_ensure_token().await`，可能刷新 token、reload 源文件、禁用凭据并 `continue`
     下一轮。新设计在 reserve 时已 `active += 1` 占槽，所以**每个失败分支
     （`continue` / `report_refresh_failure` / `report_refresh_token_invalid`）之前
     都必须先 `drop(guard)`**，否则该号槽位永久泄漏。junnl 原版正是这么做的
     （[junnl token_manager.rs:1400-1402](../../junnl/src/kiro/token_manager.rs#L1400)
     先 `drop(guard)` 再报告失败）。粘性场景失败时还应 `clear_session_affinity`。
   - priority 模式去掉“优先复用 current_id”路由，统一走 reserve_best；current_id
     仅保留给 UI“当前”标记。
   - ⚠️ **current_id 语义需明确（影响 UI，不止测试）**：UI 用
     `isCurrent` 显示“活跃”徽章（[credential-card.tsx:421](../admin-ui/src/components/credential-card.tsx#L421)、
     快照字段 [types.rs:37](../src/admin/types.rs#L37)）。去掉 priority 路由后，
     决定：**`current_id` 统一更新为“最近一次被选中的账号”**（reserve 成功即更新），
     priority/balanced 都如此。这样 UI“活跃”= 最近选中号，含义一致、不误导。
     不要让 balanced 下 current_id 停留在初始号。
   - ⚠️ **现存 current_id 断言需同步 review**：kiro.rs 已有测试断言 current_id
     行为（如 [token_manager.rs:3657](../src/kiro/token_manager.rs#L3657)
     `assert_eq!(snapshot.current_id, 2)`）。去掉路由后这些断言可能不再成立，
     必须在阶段一一并调整，否则 `cargo test` 变红。

7. `CallContext` 新增 `session_affinity_hit: bool`（仅日志展示）。

8. 常量：
   ```rust
   const SESSION_AFFINITY_TTL: StdDuration = StdDuration::from_secs(2 * 60 * 60);
   const CONCURRENCY_WAIT_POLL: StdDuration = StdDuration::from_millis(80);
   const CONCURRENCY_WAIT_BUDGET: StdDuration = StdDuration::from_secs(2);
   const STICKY_MAX_WAITING: u32 = 2;
   ```

## 阶段三：会话粘性 + guard 透传

文件：`src/kiro/provider.rs`、`src/anthropic/stream.rs`、`src/anthropic/handlers.rs`、

### provider.rs

1. `KiroCallResult` 新增：
   ```rust
   pub session_affinity_hit: bool,
   pub slot_guard: ConcurrencyGuard,
   ```
2. session key 提取（**来源有优先级，且必须由 handler 算好传入，provider 不能自解析**）：

   ⚠️ **接线关键**：provider 收到的 `request_body` 是序列化后的 `KiroRequest`
   （[handlers.rs:690-696](../src/anthropic/handlers.rs#L690)），只含 `conversation_state`
   / `profile_arn` / `additional_model_request_fields`——**原始
   `MessagesRequest.metadata.user_id` 已丢失**。所以 provider 无法自己从 request_body
   提取 metadata session，只能拿到 conversationId。

   正确做法：**在 handler / websearch_loop 里（此时还持有原始 `payload: MessagesRequest`）
   先算好 `session_key`，作为新参数往下传**。优先级：
   1. **优先** `MessagesRequest.metadata.user_id` 里的客户端 session 标识
      （Claude Code 格式含 `_session_<uuid>`）——复用
      [cache_metering.rs:553 `extract_session_id`](../src/anthropic/cache_metering.rs#L553)
      的同一套提取逻辑，保证跨轮稳定。
   2. **兜底** 转换后 `conversationState.conversationId`（无 metadata session 时，可在
      handler 从 `conversion_result.conversation_state` 或转换后 body 取）。

   实现要点：
   - 新增共享 helper（如 handler 内 `extract_session_key(payload, conversation_id)`）：
     先试 metadata，再退 conversationId。
   - **`extract_session_id` 当前是私有 `fn`（cache_metering.rs:553），复用前需改成
     `pub(crate)`**，或抽到公共模块共享；否则跨模块引用编译不过。
   - `call_api` / `call_api_stream` / `call_api_with_retry` **新增 `session_key:
     Option<&str>` 参数**，由 handler/websearch_loop 传入；provider 内部不再解析 metadata。
     这会改这几个函数签名及其全部调用方（流式/非流式/缓冲流式/websearch_loop）。
3. 重试循环 `call_api_with_retry`：
   - `acquire_context_for_session(model, group, session_key) -> (ctx, slot_guard)`，
     slot_guard 为**循环内局部变量**。
   - 任意 `continue`（429/402/网络错误）时局部 slot_guard 自动 drop，释放旧号槽位。
   - 成功时 slot_guard 移入 `KiroCallResult` 透传。
   - `CONCURRENCY_BUSY` 直接 `return Err(e)`，不再重试（内部已等过 2s）。
   - **不改现有 429/402/401 判断与退避逻辑**。
4. `call_mcp`（WebSearch/MCP 路径）不做粘性、不透传 guard：阶段一已将其改为
   `let (ctx, _guard) = ...`，本地持有 `_guard` 到该次请求结束即可，此处无需再动。

mixed WebSearch loop 的 `run_round()` 调 `provider.call_api_stream()`，随后自己

#### ⚠️ maxConcurrency 字段透传（别只加 set API，否则编辑/导入导出会丢字段）

`max_concurrency` 是持久化字段，除了单独的 set API，还要保证它在所有凭据进出路径
里不被丢弃。需逐一确认/补全：

- `AddCredentialRequest`（[types.rs:106](../src/admin/types.rs#L106)）：新增凭据时
  接受可选 `maxConcurrency`（不传默认 0）。
- `UpdateCredentialRequest`（[types.rs:208](../src/admin/types.rs#L208)）：编辑凭据时

## 槽位生命周期（最关键）

```
acquire_context → 同锁内 active += 1 → 返回 (ctx, guard)
  → provider 发上游请求
    → 成功：guard 塞进 KiroCallResult
      → handler：
          非流式：_slot_guard 持有到 body 读完 → drop
          流式：guard 存进 StreamContext → 流结束/断开/出错时 drop
          缓冲流式：guard 存进 BufferedStreamContext → 同上
    → 失败 continue：局部 guard drop，释放旧号槽位
```

## 风险控制

1. 槽位泄漏 → 必须 RAII guard，禁止手动 `active -= 1`。
2. 流式释放过早 → guard 必须活到流结束，不能只活到 provider 返回。
3. 锁顺序死锁 → entries 锁内不 await；满载等待先释放锁再 sleep；guard drop 只取
   entries 锁。
4. priority 语义破坏 → 先锁定最高可用档再档内 least-active。
5. 粘性与 least-active 冲突 → 粘性命中优先；仅绑定号满载且等待失败才放弃亲和。
6. 与风控冷却 / 分组隔离结合 → 候选过滤同时排除 disabled + throttled + 分组不匹配
   + opus 不支持，不漏 kiro.rs 现有逻辑。
7. acquire 内刷新失败泄漏 → `try_ensure_token` 每个失败分支前先 `drop(guard)`（见
   阶段一 Q3）。
8. WebSearch 释放过早 → websearch_loop 持有 guard 到 decode 完成（见阶段三）。
9. 粘性 key 每轮失效 → session key 优先取 metadata session、再退 conversationId
   （见阶段三 Q2）。

## 实现要点（评审采纳的工程约束）

- **`ConcurrencyGuard: Send + Sync`**：guard 要放进响应/stream 状态里跨 async 边界
  移动，必须 Send + Sync。它只含 `Weak<MultiTokenManager>` + `u64` + `bool`，
  `MultiTokenManager` 本就在 `Arc` 里跨 handler 共享（已 Send+Sync），故 guard 自动
  满足；写一个编译期断言 `fn _assert(){ fn is<T:Send+Sync>(){} is::<ConcurrencyGuard>() }`
  锁死该期望。
- **noop guard 不静默退化**：`make_guard` 在 self_weak 未初始化时返回 noop（drop 不
  释放），等于无并发控制。Drop/构造处应 `warn`，并加一个测试覆盖 `main.rs` 的
  `init_weak_self()` 确实被调用（reserve 后 drop，active 能归零），避免漏初始化导致
  并发控制静默失效。
- **`CONCURRENCY_BUSY` 文案固定成常量**：`map_provider_error` 用字符串 `contains`
  匹配可行，但应把标记串抽成常量（如 `const CONCURRENCY_BUSY_TAG: &str =
  "CONCURRENCY_BUSY";`），bail 和匹配共用，避免日后改中文提示导致映射失效。
