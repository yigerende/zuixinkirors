# 真实缓存命中率 90%+ 优化方案

## 目标

在不修改、不依赖模拟缓存逻辑的前提下，优化 `kiro.rs` 原有真实缓存计量路径，让正常业务流量在低并发和高并发下都能稳定获得高缓存命中率。

目标口径：

- 概览页「Token 使用趋势」缓存命中率公式保持不变：`cache_read / (input + cache_read)`。
- 优化对象是 `src/anthropic/cache_metering.rs` 的真实缓存读写推导，不改 `cache_rewriter.rs`、模拟缓存页面、模拟缓存返回改写逻辑。
- 对下游返回、请求日志、内部统计的真实缓存字段口径保持一致。
- 在连续多轮会话、长上下文、真实 Claude Code / new-api / sub2api 调用场景中，预热后缓存命中率目标达到 90% 以上。

必须说明的边界：

- 第一轮全新会话天然没有历史前缀可读，必然是写缓存或低命中。
- 如果所有请求都是完全不同用户、不同会话、不同上下文，真实缓存不可能凭空命中。
- 因此 90%+ 的目标定义为：同一用户同一会话的多轮长上下文请求，在缓存 TTL 内、缓存容量充足、session 识别正确时，后续轮次应大比例读缓存。

## 当前真实缓存链路

真实缓存字段不是上游 Kiro 返回的字段，而是 `kiro.rs` 本地推导出来的。

链路如下：

1. 下游 `new-api-dev` / `sub2api` 使用 `kiro.rs` 生成的客户端 Key 请求 `/v1/messages`。
2. `kiro.rs` 鉴权得到 `key_id`。
3. 请求进入时调用：
   - `src/anthropic/handlers.rs`
   - `compute_cache_usage(cache, &payload, key_ctx.key_id)`
4. `cache_metering.rs` 根据 `tools`、`system`、历史 `messages` 构造递增前缀段。
5. 每个前缀段通过哈希链写入内存缓存。
6. 下一轮请求查找最深命中段：
   - 命中段累计 token = `cache_read`
   - 最深段总覆盖 token - `cache_read` = `cache_creation`
7. 上游返回后，按真实 total input 做互斥分摊：
   - `input + cache_creation + cache_read == total_input_like_tokens`
8. 请求日志、统计、概览图表读取这些真实字段。

## 当前命中率低的主要原因

### 1. metadata.user_id 新格式未被真实缓存识别

`sub2api` 和现代 Claude Code 会使用新格式：

```json
{"device_id":"...","account_uuid":"...","session_id":"..."}
```

`sub2api` 已经支持两种格式：

- 老格式：`user_<device>_account_<account>_session_<uuid>`
- 新 JSON 格式：`{"device_id":"...","account_uuid":"...","session_id":"..."}`

但 `kiro.rs` 的真实缓存隔离当前只识别老格式：

```rust
user_id.split_once("_session_")
```

结果：

- JSON 格式的 `metadata.user_id` 虽然传到了 `kiro.rs`。
- `converter.rs` 能解析 JSON，并用 `session_id` 作为上游 `conversationId`。
- 但 `cache_metering.rs` 不能解析 JSON，缓存隔离退回 `key:<key_id>`。
- 多用户共用同一个客户端 Key 时，所有人的真实缓存段挤在同一个 key 维度里。
- 高并发长上下文下，互相污染、互相淘汰，导致 `cache_creation` 异常高、`cache_read` 偏低。

这是第一优先级问题。

### 2. 全局缓存容量过小

当前 `CacheMeter` 默认容量是全局 4096 个前缀段，不是每个用户/会话 4096。

高并发下，一个长上下文请求可能写入多个前缀段。几千并发、几万 RPM 时，4096 很容易被快速挤爆，导致还没到 5 分钟 TTL 就被 LRU 淘汰。

结果：

- 同会话后续请求本该命中历史前缀。
- 但前缀段已经被其他会话挤掉。
- 命中率在高并发下明显下降。

### 3. 并发首轮/相邻轮可能重复写

如果同一会话短时间内并发进来多个相似长上下文请求，当前逻辑是每个请求独立计算、独立 lookup、独立 write。

可能出现：

- 请求 A 还没写入最深前缀。
- 请求 B 同时 lookup，看不到 A 的结果。
- A/B 都记录为写缓存。

这会放大 `cache_creation`，降低命中率。

### 4. 前缀签名对部分动态字段仍可能过敏

真实缓存命中依赖“历史前缀稳定”。只要进入哈希签名的内容发生变化，就会被认为不是同一前缀。

需要重点检查：

- system 中是否有每次变化的时间、环境、随机 ID。
- tools schema 顺序是否稳定。
- message content 是否包含每轮变化但不影响复用的元信息。
- thinking、tool_result、image 等 block 是否使用稳定签名。

当前实现已经对部分字段做了过滤，但仍需要增加观测来证明 miss 是 session 问题、容量问题，还是前缀不稳定问题。

## 总体方案

真实缓存要稳定 90%+，核心不是改返回字段，而是让缓存键做到：

- 会话隔离准确。
- 同会话多轮稳定。
- 高并发不互相挤爆。
- 相同前缀不重复写。
- miss 原因可观测。

方案分五层实施。

## 第一层：统一 session 解析

### 目标

让 `cache_metering.rs`、`converter.rs`、`handlers.rs` 使用同一套 `metadata.user_id` session 提取逻辑。

### 改法

新增或改造一个统一函数，例如：

```rust
pub(crate) fn extract_metadata_session_id(user_id: &str) -> Option<String>
```

识别顺序：

1. JSON 格式：

```json
{"device_id":"...","account_uuid":"...","session_id":"..."}
```

取 `session_id`。

2. 老格式：

```text
user_<device>_account_<account>_session_<uuid>
```

取 `_session_` 后面的 UUID。

3. 兼容宽松格式：

```text
session_<uuid>
```

只取合法 UUID 部分，避免把后续脏字符也带入 seed。

### 影响范围

应该只改真实缓存/session 粘性相关函数：

- `src/anthropic/cache_metering.rs`
  - `isolation_seed`
  - `extract_session_id`
- `src/anthropic/handlers.rs`
  - `compute_session_key`
- 可选择让 `converter.rs` 复用统一函数，避免两套解析逻辑长期分叉。

注意：真实缓存的隔离 seed 不应使用 `converter.rs` 在缺少 metadata 时随机生成的
`conversationId`。随机 `conversationId` 每次不同，会让缓存永远 miss。只有当请求本身携带
稳定会话标识，或 converter 能从 `metadata.user_id` 解析出稳定 session 时，才适合进入
真实缓存 seed。

不改：

- `src/anthropic/cache_rewriter.rs`
- Admin UI 模拟缓存页面
- 模拟缓存配置结构
- 模拟缓存返回改写逻辑

### 预期效果

同一个 `metadata.user_id.session_id` 的多轮请求会进入同一个真实缓存命名空间：

```text
sess:<session_id>
```

不同用户、不同会话不会互相污染。

## 第二层：提升缓存容量并按会话分片

### 目标

高并发下不因为全局 4096 段太小导致真实缓存刚写入就被挤掉。

### 建议配置

将缓存容量从固定常量改为配置项：

```json
{
  "cacheMetering": {
    "maxEntries": 200000,
    "defaultTtlSeconds": 300,
    "maxSessionEntries": 2048
  }
}
```

推荐默认值：

- 小规模：`maxEntries = 50000`
- 中等规模：`maxEntries = 200000`
- 高并发生产：`maxEntries = 500000` 或更高
- `defaultTtlSeconds = 300` 保持 5 分钟
- `maxSessionEntries = 1024~4096`，防止单个异常会话挤爆全局

说明：项目配置结构当前使用 `#[serde(rename_all = "camelCase")]`，因此 `config.json`
里建议使用 `cacheMetering`、`maxEntries` 这类 camelCase 字段；Rust 结构体内部可以继续
使用 `cache_metering`、`max_entries` 命名。

### 数据结构建议

当前全局 LRU 可以改为：

```text
global map: segment_hash -> cache entry
session index: session_seed -> segment_hash list
expiry wheel / heap: expire_at -> segment_hash
```

淘汰策略：

1. 先删除过期项。
2. 全局超限时按 LRU 删除。
3. 单 session 超限时只淘汰该 session 最老段。

### 预期效果

高并发不同会话之间不再因为一个 key 维度混杂而互相冲掉；即使不同会话很多，也能通过更大的容量和 per-session 限制保持稳定。

## 第三层：同会话并发 singleflight

### 目标

减少同一会话相同/相近前缀在高并发下重复写缓存。

### 设计

给最深前缀 hash 增加短生命周期的 in-flight 状态：

```text
inflight_key = session_seed + deepest_segment_hash
ttl = 5~15 seconds
```

当请求 lookup 发现 miss 时：

1. 如果没有 in-flight：
   - 当前请求成为 writer。
   - 正常记录 `cache_creation`。
   - 完成后写入缓存段。
2. 如果已有 in-flight：
   - 当前请求等待极短时间，例如 20~100ms。
   - 等待后再次 lookup。
   - 如果命中，则记录为 `cache_read`。
   - 如果仍未命中，再按 miss 处理。

### 注意

不能为了命中率无限阻塞请求。singleflight 只是降低并发重复写，不影响正常响应。

### 预期效果

同一会话并发请求不会全部同时记成 `cache_creation`，高并发下写缓存峰值下降，读缓存比例上升。

## 第四层：前缀签名稳定性治理

### 目标

确保真实可复用的上下文不会因为无意义动态字段导致 hash 变化。

### 需要检查和固定的字段

1. `system`
   - 保留真实 system 文本。
   - 对已知每轮变化但不影响上下文复用的 wrapper 元信息做可选归一化。

2. `tools`
   - 工具按 name 排序后参与签名，或保持客户端原顺序但记录顺序变化 miss。
   - schema 序列化必须稳定。

3. `messages`
   - 最后一条 user 不切段，这个逻辑保持。
   - 历史 assistant/user 内容参与前缀。
   - 对 tool_result 中每轮变化的非语义字段，需要评估是否过滤。

4. image
   - base64 大字段可以用内容 hash 参与签名。
   - token 估算继续保留。

5. thinking
   - 如果 thinking/signature 每轮变化且不影响 prompt cache，应避免作为可复用前缀的强签名字段。

### 观测优先

不建议一上来大幅过滤字段。先加 miss reason：

```text
miss_reason = session_parse_failed | cold_session | expired | evicted | prefix_changed | capacity | inflight
```

用数据确认是哪类问题后，再决定过滤哪些字段。

## 第五层：观测、统计和压测

### 新增内部指标

真实缓存应至少记录：

- `cache_meter.session_parse_ok`
- `cache_meter.session_parse_failed`
- `cache_meter.seed_source`
  - `metadata_json`
  - `metadata_legacy`
  - `key_id`
- `cache_meter.lookup_hit`
- `cache_meter.lookup_miss`
- `cache_meter.evicted_lru`
- `cache_meter.evicted_expired`
- `cache_meter.evicted_session_limit`
- `cache_meter.entries_total`
- `cache_meter.sessions_total`
- `cache_meter.inflight_wait`
- `cache_meter.inflight_hit_after_wait`

### 请求日志建议增加调试字段

可只在 debug 或采样日志里输出：

```json
{
  "cache_seed_source": "metadata_json",
  "cache_session_fp": "sha1(session_id)[0..8]",
  "cache_segments": 12,
  "cache_deepest_hit_depth": 10,
  "cache_miss_reason": "evicted_lru"
}
```

注意不要明文写完整 `metadata.user_id` 或完整 session id。

### 验证用例

必须覆盖以下端到端场景：

1. 老格式 metadata：
   - 第一轮写缓存。
   - 第二轮同 session 读缓存。
   - 不同 session 不互相命中。

2. JSON 格式 metadata：
   - 第一轮写缓存。
   - 第二轮同 `session_id` 读缓存。
   - 不同 `session_id` 不互相命中。

3. 同一个客户端 Key 多用户：
   - user A session A 命中 A。
   - user B session B 不命中 A。
   - A/B 并发不互相污染。

4. 换凭据/换上游账号：
   - 只要下游 `metadata.user_id.session_id` 不变，真实缓存仍按同会话命中。
   - 账号切换不应导致本地真实缓存重新按 key 写一遍。

5. 高并发：
   - 1000 并发，不同 session 长上下文。
   - 1000 并发，同 session 相邻轮。
   - 观察 LRU 淘汰、singleflight 效果、命中率。

6. TTL：
   - 5 分钟内同 session 命中。
   - TTL 后自然 miss 并重新写。

7. 三种返回路径：
   - 非流式。
   - 流式。
   - buffered stream。

所有测试均应关闭模拟缓存。

## Admin 可视化配置页面

建议在 Admin 顶部菜单新增一个独立页面：

```text
真实缓存
```

它和现有「模拟缓存」页面并列，但两者职责完全分开：

- 「真实缓存」：配置 `cache_metering.rs` 的真实缓存推导、容量、TTL、session 识别、singleflight、观测。
- 「模拟缓存」：继续只配置最终返回给下游的模拟缓存 usage 改写。
- 两个页面不共用字段，不互相读取配置，不互相改变行为。

### 顶部菜单设计

当前菜单已有：

```text
概览 / 凭据管理 / 客户端 Key / 分组管理 / 模拟缓存 / 请求日志
```

建议新增：

```text
概览 / 凭据管理 / 客户端 Key / 分组管理 / 真实缓存 / 模拟缓存 / 请求日志
```

移动端短标签：

```text
真实
```

图标建议：

- `DatabaseZap`
- `Gauge`
- `Activity`
- `Settings2`

如果 `Gauge` 已被「模拟缓存」占用，真实缓存页建议用 `DatabaseZap` 或 `Settings2`，避免用户混淆。

### 页面模块 1：运行状态

用于快速判断真实缓存是否健康，不直接改规则。

展示字段：

- 当前缓存开关状态。
- 当前真实缓存条目数：`entries_total`。
- 当前 session 数：`sessions_total`。
- 当前全局容量上限：`maxEntries`。
- 当前默认 TTL：`defaultTtlSeconds`。
- LRU 淘汰数：`evicted_lru`。
- 过期淘汰数：`evicted_expired`。
- 单 session 超限淘汰数：`evicted_session_limit`。
- session 解析成功/失败数量。
- 最近 5 分钟真实缓存命中率。
- 最近 5 分钟 `cache_read`、`cache_creation`、`input`。

推荐加一个只读提示：

```text
该页面显示真实缓存计量状态，不显示模拟缓存改写后的字段。
```

### 页面模块 2：基础配置

这些配置适合做成页面表单，并写入 `config.json`：

```json
{
  "cacheMetering": {
    "enabled": true,
    "maxEntries": 200000,
    "defaultTtlSeconds": 300,
    "persistEnabled": true,
    "persistIntervalSeconds": 60
  }
}
```

字段说明：

- `enabled`
  - 是否启用真实缓存计量。
  - 默认开启。
  - 关闭后 `cache_read/cache_creation` 都会失去真实推导能力，不建议生产关闭。

- `maxEntries`
  - 全局最大缓存段数。
  - 默认建议 `200000`。
  - 高并发生产建议 `500000` 或更高。
  - 页面需要显示内存风险提示。

- `defaultTtlSeconds`
  - 默认 TTL。
  - 默认 `300`，即 5 分钟。
  - 可配置范围建议 `60~3600`。

- `persistEnabled`
  - 是否落盘持久化 `cache_metering.json`。
  - 默认开启。

- `persistIntervalSeconds`
  - 持久化间隔。
  - 默认 `60`。
  - 太小会增加 IO，太大则重启丢失更多热缓存。

这些字段中：

- `enabled`、`maxEntries`、`defaultTtlSeconds` 可支持热更新。
- `persistEnabled`、`persistIntervalSeconds` 可先设计为保存后重启生效，降低实现复杂度。

### 页面模块 3：Session 识别配置

这些配置对应第一层“统一 session 解析”。

```json
{
  "cacheMetering": {
    "session": {
      "enableJsonMetadata": true,
      "enableLegacyMetadata": true,
      "fallbackToKeyId": true,
      "strictUuid": true
    }
  }
}
```

字段说明：

- `enableJsonMetadata`
  - 识别 `metadata.user_id` 的 JSON 格式。
  - 必须默认开启。
  - 这是当前命中率问题的核心修复项。

- `enableLegacyMetadata`
  - 识别老格式 `_session_<uuid>`。
  - 默认开启，兼容旧 Claude Code。

- `fallbackToKeyId`
  - 最后兜底到 `key:<key_id>`。
  - 默认开启。
  - 页面提示：高并发多用户共用 Key 时，长期 fallback 到 key_id 会导致命中率低和缓存污染。

- `strictUuid`
  - 只接受合法 UUID session。
  - 默认开启。
  - 避免脏字符串导致 seed 不稳定。

页面应展示 seed source 分布：

```text
metadata_json / metadata_legacy / key_id
```

不要把 `converter.rs` 在缺少 metadata 时临时生成的随机 `conversationId` 当作真实缓存 seed。
这个 ID 每次请求都可能不同，用它做 seed 会导致同一真实会话永远无法稳定命中。

如果 `key_id` 占比过高，页面应显示告警：

```text
大量请求退回 key_id 隔离，可能导致多用户缓存互相污染，请检查 metadata.user_id 是否正确传递和解析。
```

### 页面模块 4：容量与淘汰策略

这些配置对应第二层“提升缓存容量并按会话分片”。

```json
{
  "cacheMetering": {
    "eviction": {
      "maxSessionEntries": 2048,
      "cleanupIntervalSeconds": 30,
      "evictExpiredFirst": true
    }
  }
}
```

字段说明：

- `maxSessionEntries`
  - 单 session 最大段数。
  - 默认建议 `2048`。
  - 防止单个超长会话挤爆全局缓存。

- `cleanupIntervalSeconds`
  - 过期清理间隔。
  - 默认 `30`。

- `evictExpiredFirst`
  - 超限时优先清理过期项。
  - 默认开启。

页面展示：

- 全局使用率：`entries_total / maxEntries`。
- Top session 占用排行，只显示 session 指纹，不显示完整 session。
- 淘汰原因分布。

### 页面模块 5：并发去重 singleflight

这些配置对应第三层“同会话并发 singleflight”。

```json
{
  "cacheMetering": {
    "singleflight": {
      "enabled": true,
      "waitMs": 50,
      "inflightTtlSeconds": 10,
      "maxInflight": 10000
    }
  }
}
```

字段说明：

- `enabled`
  - 是否开启同 session 同前缀并发去重。
  - 默认建议开启。

- `waitMs`
  - miss 后等待已有 writer 的时间。
  - 默认 `50ms`。
  - 建议范围 `0~200ms`。

- `inflightTtlSeconds`
  - in-flight 标记存活时间。
  - 默认 `10`。

- `maxInflight`
  - 最大并发 in-flight 记录数。
  - 默认 `10000`。

页面展示：

- `inflight_wait`
- `inflight_hit_after_wait`
- `inflight_timeout`
- singleflight 后命中率提升估计

页面提示：

## 分阶段实施计划

### 阶段 1：修 session 解析

优先级最高，改动最小。

任务：

- 让 `cache_metering.rs` 支持 JSON `metadata.user_id.session_id`。
- 让 `handlers.rs::compute_session_key` 复用同一解析函数。
- 增加 JSON 格式单元测试。
- 增加同 key 不同 JSON session 隔离测试。

验收：

- JSON metadata 下，同 session 第二轮 `cache_read > 0`。
- JSON metadata 下，不同 session `cache_read == 0`。
- 关闭模拟缓存时三种接口路径统计一致。

### 阶段 2：容量配置化

任务：

- 将 `DEFAULT_CAPACITY = 4096` 改成配置项。
- 默认值提高到更适合生产的容量。
- 保留老配置缺省兼容。
- 启动日志打印当前容量、TTL、持久化路径。
- Admin 顶部菜单新增「真实缓存」页面，至少支持查看和保存容量/TTL/session 解析配置。

验收：

- 高并发压测时 `evicted_lru` 明显下降。
- 同 session 5 分钟内不因容量过小被提前挤掉。
- 页面保存后配置写入 `config.json`，运行时生效字段无需重启即可生效。

### 阶段 3：观测 miss reason

任务：

- 增加 seed source。
- 增加 miss reason。
- 增加 debug 采样日志。
- Admin「真实缓存」页面展示 seed source、miss reason、淘汰原因和最近命中率。

验收：

- 能明确区分是 cold、expired、evicted、prefix_changed、session_parse_failed。
- 线上出现命中率低时，可以从日志定位原因。
- 页面可以直观看到是否大量请求 fallback 到 `key_id`。

### 阶段 4：singleflight

任务：

- 增加同 session + deepest segment 的短期 in-flight 表。
- miss 时短等待一次再 lookup。
- 等待时间和开关配置化。

验收：

- 同 session 并发长上下文时，重复 cache_creation 降低。
- P95 延迟增加可控。

### 阶段 5：前缀稳定性优化

任务：

- 基于 miss reason 找出真正导致 prefix_changed 的字段。
- 只对确认无语义影响的动态字段做归一化。
- 增加回归测试，保证不同真实上下文不会误命中。

验收：

- prefix_changed 占比下降。
- 无跨会话、跨上下文误命中。

## 90%+ 命中率达成路径

在真实业务中，命中率主要由下面因素决定：

```text
命中率 = cache_read / (input + cache_read)
```

要达到 90%+，需要让大部分长上下文 token 出现在 `cache_read`，而不是 `input` 或 `cache_creation`。

达成路径：

1. JSON session 正确识别，避免退回 `key_id`。
2. 同一会话多轮请求进入同一个 `sess:<session_id>` seed。
3. 不同用户/不同会话互相隔离，避免污染。
4. 缓存容量足够大，5 分钟内不被提前淘汰。
5. 高并发同前缀请求通过 singleflight 降低重复写。
6. 前缀签名稳定，避免无意义动态字段导致 miss。

这样在典型 Claude Code 长上下文多轮会话里：

- 第一轮：低命中或写缓存。
- 第二轮起：历史长上下文应主要进入 `cache_read`。
- 会话持续越久、上下文越长，`cache_read / (input + cache_read)` 越接近 90%+。

## 不做的事情

本方案明确不做：

- 不改模拟缓存页面。
- 不改模拟缓存命中规则。
- 不靠最终返回字段强行伪造真实统计。
- 不把所有 miss 都改成 read。
- 不让不同 session 互相命中。
- 不为了命中率牺牲真实隔离和统计可信度。

## 推荐第一步落地

先只做阶段 1：

```text
统一 metadata.user_id session 解析，支持 JSON 格式。
```

这是当前最明确、收益最高、风险最低的问题点。它不触碰模拟缓存，也不改变 cache token 的计算公式，只修正真实缓存隔离 seed。

阶段 1 完成后，再用真实 `new-api-dev` / `sub2api` 请求做端到端测试。如果命中率仍低，再根据 miss reason 推进容量、singleflight 和前缀稳定性治理。
