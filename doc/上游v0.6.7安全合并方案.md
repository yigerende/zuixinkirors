# 上游 v0.6.7 安全合并方案

## 目标

将上游 `ZyphrZero/kiro.rs` 的最新版本 `v0.6.7` 中有价值的修复和功能合并到当前二开项目，同时保留本项目已有的真实缓存、模拟缓存、缓存配置页面、请求日志、账号调度和统计逻辑。

本次合并不建议直接执行 `git merge zyphrzero/master`。当前项目已经有较多二开内容，直接合并会产生大量冲突，并可能删除或覆盖本地新增的缓存相关文件。

合并前必须先确认工作区状态。除本次准备提交的文档或合并分支改动外，不应带着其它未提交代码直接开始移植，避免把旧调试改动、半成品改动一起混入上游合并。

## 为什么不能直接 merge

本地项目相对上游已经新增和修改了大量二开内容，包括：

- 真实缓存命中率推理逻辑
- 模拟缓存页面和模拟缓存控制逻辑
- 缓存配置页面
- 请求日志和概览统计增强
- 账号调度、并发、亲和度相关逻辑
- 多份本地文档和启动脚本

直接合并上游会看到上游视角下的删除项，例如：

- `admin-ui/src/components/cache-metering-page.tsx`
- `admin-ui/src/components/cache-optimizer.tsx`
- `admin-ui/src/hooks/use-cache-metering.ts`
- `admin-ui/src/hooks/use-cache-optimizer.ts`
- `src/anthropic/cache_rewriter.rs`
- 多份 `doc/*.md`

这些不是应该删除的文件，而是本地二开新增内容。直接 merge 会有误删和覆盖风险。

## 推荐合并策略

采用“分批手工移植”的方式，不做整包 merge。

每一批只合并一个主题，合并后立即编译和测试。这样出现问题时可以快速定位是哪一批引入的。

手工移植时以本地二开代码为主，把上游新增逻辑嵌进去；不要用上游文件整段覆盖本地文件。尤其是请求链路、缓存链路、账号调度链路和管理后台页面，必须逐段对照。

## 第一批：后端稳定性修复

优先合并这部分，因为它们风险相对可控，且对正式环境稳定性有帮助。

| 上游能力 | 涉及文件 | 合并方式 | 作用 |
|---|---|---|---|
| 凭据 ID 单调递增 | `src/kiro/token_manager.rs` | 手工移植 | 删除账号后新增账号不会复用旧 ID，避免新账号继承旧账号 trace/usage 历史 |
| 凭据写盘锁 | `src/kiro/token_manager.rs` | 手工移植 | 并发导入或并发修改凭据时，避免整文件写盘互相覆盖 |
| 并发添加凭据二次去重 | `src/kiro/token_manager.rs` | 手工移植 | 多个并发请求添加同一凭据时，只允许成功插入一条 |
| Opus 账号支持过滤修复 | `src/kiro/token_manager.rs` | 手工移植 | 当前账号优先逻辑不能绕过 Opus 订阅能力过滤 |
| 删除凭据清理 trace | `src/admin/service.rs`、`src/admin/trace_db.rs` | 手工移植 | 删除账号后清理相关请求记录，避免统计继续挂在旧 ID 上 |

删除凭据清理 trace 按上游方案合并即可。删除账号后同步删除该凭据关联的请求记录和失败统计，避免后续新增账号被旧 `credential_id` 的历史数据污染。

第一批测试：

```powershell
cargo fmt
cargo check
cargo test token_manager
cargo test trace_db
```

需要人工验证：

- 删除一个凭据后再新增凭据，新凭据 ID 不复用被删 ID。
- 并发添加相同 API Key，只会新增一条。
- 删除凭据后，请求日志/凭据统计不再显示该凭据的旧失败数据。
- Opus 请求不会选到不支持 Opus 的账号。

| 上游能力 | 涉及文件 | 合并方式 | 作用 |
|---|---|---|---|

## 不建议合并的内容

以下内容不建议直接从上游覆盖：

| 内容 | 原因 |
|---|---|
| 整个 `admin-ui/src/components/dashboard.tsx` | 本地二开逻辑多，直接覆盖风险很高 |
| 整个 `src/kiro/token_manager.rs` | 本地账号调度、并发、亲和度逻辑重要，不能被覆盖 |
| 整个 `src/anthropic/handlers.rs`、`stream.rs` | 本地缓存读写、最终返回兜底、统计链路都在请求路径附近 |
| 删除本地新增的缓存相关文件 | 这些是本项目核心二开能力 |
| 直接采用上游 `doc` 状态 | 会删除本地项目文档 |

## 推荐操作流程

创建专门合并分支：

```powershell
git status --short
git switch -c merge-upstream-v0.6.7-safe
```

如果 `git status --short` 里存在非本次合并相关的代码改动，应先提交、暂存或退回这些改动，再开始移植。

确认上游 ref：

```powershell
git fetch https://github.com/ZyphrZero/kiro.rs.git master:refs/remotes/zyphrzero/master --tags
git log --oneline HEAD..zyphrzero/master
```

每批合并完成后提交一次：

```powershell
cargo fmt
cargo check
git status --short
git add <本批修改文件>
git commit -m "merge upstream v0.6.7 credential stability fixes"
```

前端批次合并完成后：

```powershell
cd admin-ui
npm run build
```

## 最终建议

本次上游 `v0.6.7` 建议按以下顺序合并：

1. 后端稳定性修复。
2. effort 分级归一化。
3. 远程 Social 登录。
4. 批量导入和批量验活优化。
5. 凭据管理和仪表盘 UI 优化。

其中第 1、2 批最值得优先合并；第 3 批按远程登录需求决定；第 4、5 批前端影响较大，建议等后端稳定后再做。

整个过程必须坚持手工移植和分批测试，避免直接整包合并导致本地二开能力被覆盖。
