# kiro.rs 并发调度中心迁移方案

## 目标

把 junnl 已验证的并发调度中心迁移到 kiro.rs，用来解决当前 `balanced` 模式在高并发下按历史成功次数选号导致的账号踩踏问题。

迁移边界：

- 只改账号调度、并发槽位、会话粘性、管理面板展示与设置。
- 不改 provider 内部请求转换、模拟缓存、usage 统计、请求日志真实字段、429/402/401 失败判断语义。
- 并发计数只做单实例内存版，不上 Redis。

## 当前 kiro.rs 现状

kiro.rs 当前调度核心在 `src/kiro/token_manager.rs`。

现有 `priority` 模式：

- 启动时选 `priority` 最小的可用账号作为 `current_id`。
- 每次请求优先复用 `current_id`。
- 当前账号不可用、禁用、冷却、分组不匹配时，才重新选优先级最高账号。
- 没有当前在途请求数概念。

现有 `balanced` 模式：

- 每次请求重新选择 `success_count` 最少的可用账号。
- 平手时按 `priority` 最小。
- `success_count` 只在上游成功返回后才加 1。
- 高并发同时进来时，请求会看到同一份旧的 `success_count`，容易一起选中同一个账号。

现有缺口：

- 没有 `active_concurrency` 当前在途数。
- 没有账号级并发上限。
- 没有 RAII 槽位守卫。
- 没有真正会话粘性。
- 流式请求拿到上游响应头后就返回给 handler，真正结束在 body/stream 读完之后，所以槽位不能在 provider 返回时释放。

## junnl 方案精读结论

junnl 的并发调度中心核心在 `src/kiro/token_manager.rs`、`src/kiro/provider.rs`、`src/anthropic/handlers.rs`、`src/anthropic/stream.rs`。

它解决问题靠四件事：

1. `CredentialEntry` 里加运行时字段：
   - `active`: 当前在途请求数，选中账号时立即加 1。
   - `waiting`: 当前等待该账号释放的请求数，仅用于展示和粘性过载判断。

2. `ConcurrencyGuard`：
   - 账号被选中并 `active += 1` 后返回一个 RAII 守卫。
   - 守卫 `Drop` 时自动 `active = active.saturating_sub(1)`。
   - 所有错误、重试、客户端断开、流式结束都靠 Drop 释放，避免手写释放漏分支。

3. least-active 选号：
   - `balanced`: 先选 `active` 最小，再按 `success_count`，再平手随机。
   - `priority`: 先锁定最高优先级档，只在同优先级档内按 `active` 最小和平手随机。
   - 选号和 `active += 1` 必须在同一把 `entries` 锁内完成，避免 TOCTOU 踩踏。

4. 会话粘性：
   - 仅 `balanced` 模式启用。
   - 用 conversation/session key 绑定到上次使用的账号。
   - 绑定账号可用且未满载时直接复用。
   - 绑定账号满载时短等，超时或等待者过多就放弃亲和，换号并重绑。

## 迁移后的目标行为

### priority 模式

保持“优先用高优先级账号”的语义。

规则：

1. 先筛选可用账号：未禁用、未风控冷却、支持当前模型、分组匹配。
2. 找出 `priority` 最小的一档。
3. 在这一档内按 `active` 最小选择。
4. `active` 和长期统计都相同的情况下随机打散。
5. 只有当最高优先级档因 `maxConcurrency > 0` 全部满载被过滤掉时，才落到下一优先级档。

注意：不能写成 `(active, priority)`，否则低优先级空闲账号会抢过高优先级繁忙账号，破坏 priority 语义。

### balanced 模式

升级为真正实时负载均衡。

规则：

1. 如果存在会话粘性绑定，优先尝试绑定账号。
2. 未命中粘性或绑定失效时，在候选账号中选择：
   - `active` 最小
   - 再 `success_count` 最小
   - 再 `priority` 最小或同组随机
3. 选中同时 `active += 1` 并生成 `ConcurrencyGuard`。

### 会话粘性

kiro.rs 当前没有真正会话粘性。迁移后建议加：

- `session_affinity: Mutex<HashMap<String, SessionAffinityEntry>>`
- TTL：2 小时
- 只在 `balanced` 模式启用
- session key 来源优先用 Kiro 请求体里的 conversation id；如果 kiro.rs 已有从 Anthropic 请求转换得到的 session/user 信息，也可以在转换后传入 provider

粘性过载：

- 绑定账号未满载：继续用绑定账号，`sessionAffinityHit=true`
- 绑定账号满载：
  - 等待最多 2 秒
  - 轮询间隔 80ms
  - 等待者超过 2 个时放弃等待
  - 放弃后走普通 least-active 选号，并把会话重绑到新账号

## 关键数据结构

### CredentialEntry 新增运行时字段

```rust
struct CredentialEntry {
    // existing fields...
    active: u32,
    waiting: u32,
}
```

`active/waiting` 不持久化，只存在内存。

### KiroCredentials 新增持久化字段

```rust
#[serde(default)]
#[serde(skip_serializing_if = "is_zero")]
pub max_concurrency: u32,
```

含义：

- `0`: 不限制，默认值，兼容旧配置。
- `>0`: 该账号最大并发。`active >= max_concurrency` 时视为满载。

### CallContext 新增字段

```rust
pub session_affinity_hit: bool,
```

只给请求日志/UI展示，不参与后续业务逻辑。

### Provider 返回结果新增字段

```rust
pub struct KiroCallResult {
    pub response: reqwest::Response,
    pub credential_id: u64,
    pub session_affinity_hit: bool,
    pub slot_guard: ConcurrencyGuard,
}
```

## 槽位生命周期

这是迁移里最重要的点。

正确生命周期：

1. `token_manager.acquire_context_for_session(...)`
2. 选中账号，同一把锁内 `active += 1`
3. 返回 `(CallContext, ConcurrencyGuard)`
4. provider 发上游请求
5. 成功时把 `slot_guard` 塞进 `KiroCallResult`
6. handler 接到结果后：
   - 非流式：读完整个 response body 后 guard drop
   - 流式：guard 存进 `StreamContext`
   - 缓冲流式：guard 存进 `BufferedStreamContext`
7. 流读完、出错、客户端断开，context 被 drop，guard 自动释放槽位

错误和重试：

- provider 内部任意 `continue` 时，局部 `slot_guard` 自动 drop，释放旧账号槽位。
- 不要在 429/402/401/网络错误分支里手写释放。

## 实施顺序

### 第一阶段：只做核心调度，不做 UI

目标：先让后端能防踩踏。

1. 给 `CredentialEntry` 加 `active/waiting`
2. 加 `ConcurrencyGuard`
3. 改 `acquire_context` 返回 guard
4. 改 provider 透传 guard
5. handler/stream 持有 guard 到请求结束
6. balanced 改 least-active
7. priority 同优先级档内 least-active

验证：

- `cargo test`
- 单元测试：5 个账号 balanced 连续 reserve 5 次不释放，应分布到 5 个账号
- 单元测试：guard drop 后 active 归零
- 单元测试：priority 模式不同优先级时不越级

### 第二阶段：并发硬上限

目标：支持保护单账号不被打爆。

1. `KiroCredentials.max_concurrency`
2. 满载账号过滤
3. 全部满载时短等待
4. 超时返回 `CONCURRENCY_BUSY`
5. provider 把 `CONCURRENCY_BUSY` 映射成上游错误，让 handler 返回 429

验证：

- 单账号 `maxConcurrency=1`，占住后第二个请求等待约 2 秒后返回繁忙
- 两账号都 `maxConcurrency=1`，一个满载时选另一个

### 第三阶段：会话粘性

目标：balanced 模式同会话尽量固定账号。

1. 增加 `session_affinity`
2. 从请求体提取 conversation/session key
3. balanced 下命中绑定账号优先
4. 绑定账号失效清绑
5. 绑定账号满载短等，超时/等待者过多换号并重绑

验证：

- 同一 conversation 连续请求命中同一账号
- 绑定账号禁用后清绑并换号
- 绑定账号满载但 2 秒内释放，仍用原账号
- 绑定账号满载超时，换账号且 `sessionAffinityHit=false`

### 第四阶段：管理面板

目标：可观察、可配置。

1. 凭据列表接口返回 max/active/waiting
2. 前端展示并发状态
3. 单个设置上限
4. 批量设置上限

## 风险点

1. 槽位泄漏
   - 必须用 RAII guard，禁止手动维护 active。

2. 流式释放过早
   - guard 必须活到流结束，不能只活到 provider 返回。

3. 锁顺序死锁
   - `entries` 锁内不 await。
   - 等待满载时必须释放锁后 sleep。
   - guard drop 只拿 `entries` 锁，不嵌套其他锁。

4. priority 语义被破坏
   - priority 模式必须先锁定最高优先级档，再档内 least-active。

5. 会话粘性和 least-active 冲突
   - 粘性命中必须优先于普通 balanced 选号。
   - 只有绑定号满载且等待失败时才放弃亲和。

6. 与现有风控冷却结合
   - kiro.rs 当前已有 `throttled_until`。
   - 候选过滤必须同时排除 disabled 和 throttled。
   - junnl 原版没有完全一样的风控字段，迁移时不能漏掉 kiro.rs 现有冷却逻辑。

## 和模拟缓存/请求日志的边界

并发调度只影响“选哪个上游账号”和“什么时候释放账号槽位”。

不得影响：

- 模拟缓存返回 usage 改写
- 原始 Token / 模拟TOKEN 日志字段
- cache creation / cache read 改写
- provider 请求体转换
- trace/usage 原始统计

可新增日志字段：

- `sessionAffinityHit`
- `activeConcurrency` 快照可选
- `waitingConcurrency` 快照可选

## 推荐最终形态

优先迁移 junnl 的“轻量内存版”而不是做复杂队列：

- 单实例内存 `active/waiting`
- RAII guard 释放
- balanced least-active
- priority 同档 least-active
- 可选 `maxConcurrency`
- balanced 会话粘性
- 满载短等待 2 秒
- waiting 阈值 2

这套和当前 kiro.rs 架构最贴合，迁移成本最低，也最不容易碰坏现有模拟缓存和请求日志。
